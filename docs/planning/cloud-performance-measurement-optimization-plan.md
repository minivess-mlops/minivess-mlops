## 2. Per-Job Metrics — What to Measure for EVERY Job

### 2.1 Metric Key Schema (extends `MetricKeys`)

All new keys follow the existing slash-prefix convention from `metric_keys.py`.

```python
# --- Phase timing (logged as MLflow metrics at step=0) ---
# These measure wall-clock seconds for each phase of the job.

# Setup phase (already partially implemented in infrastructure_timing.py)
"perf/setup_total_seconds"          # Total setup phase
"perf/docker_pull_seconds"          # Docker image pull time
"perf/dvc_pull_seconds"             # DVC data pull time
"perf/hf_download_seconds"          # HuggingFace model weight download
"perf/cuda_verify_seconds"          # nvidia-smi + torch.cuda check
"perf/mlflow_verify_seconds"        # MLflow connectivity check
"perf/spot_provision_seconds"       # Time from STARTING to RUNNING

# Training phase
"perf/training_total_seconds"       # Total training wall time (all folds)
"perf/training_per_epoch_seconds"   # Average seconds per epoch
"perf/dataloader_build_seconds"     # DataLoader construction time
"perf/model_build_seconds"          # Model instantiation time

# Post-training phase (NEW — currently unmeasured)
"perf/swag_total_seconds"           # Total SWAG wall time
"perf/swag_per_epoch_seconds"       # Average SWAG epoch time
"perf/swag_bn_update_seconds"       # BatchNorm recalibration time
"perf/swag_model_rebuild_seconds"   # Time to rebuild model + DataLoader for SWAG
"perf/swag_overhead_ratio"          # swag_total / training_total (target: <2.0)

# Job-level totals
"perf/job_total_seconds"            # End-to-end wall time (setup + train + PT)
"perf/gpu_active_seconds"           # Time GPU was doing compute (train + SWAG)
"perf/gpu_idle_seconds"             # Time GPU was idle (setup, I/O, overhead)
"perf/gpu_utilization_fraction"     # gpu_active / job_total
"perf/submission_to_start_seconds"  # SkyPilot submission to first GPU compute

# Data I/O
"perf/data_volume_count"            # Number of training volumes loaded
"perf/data_total_mb"                # Total data loaded in MB
"perf/checkpoint_write_seconds"     # Time to write checkpoint to GCS

# --- Per-job tags (logged as MLflow tags, not metrics) ---
"perf/cloud_provider"               # gcp | runpod
"perf/controller_cloud"             # Where SkyPilot controller runs
"perf/gpu_type"                     # L4 | A100-80GB
"perf/spot_enabled"                 # true | false
"perf/image_size_mb"                # Docker image compressed size
"perf/region"                       # europe-north1 etc.
```

### 2.2 Derived Metrics (Computed Post-Hoc)

These are computed by the analysis flow or DuckDB queries, not logged per-job:

```
perf/setup_cost_usd         = (setup_total_seconds / 3600) * hourly_rate
perf/training_cost_usd      = (training_total_seconds / 3600) * hourly_rate
perf/swag_cost_usd          = (swag_total_seconds / 3600) * hourly_rate
perf/wasted_gpu_usd         = (gpu_idle_seconds / 3600) * hourly_rate
perf/amortization_epochs    = 9 * setup_total / training_per_epoch  (setup < 10% of total)
```

## 3. Storage Strategy — Where Metrics Live

### 3.1 Primary: MLflow (per-job, durable)

Every job already logs to MLflow. The `perf/*` metrics go into the same run:

```python
# In train_flow.py, after each phase completes:
mlflow.log_metric("perf/setup_total_seconds", setup_duration)
mlflow.log_metric("perf/training_total_seconds", training_duration)
mlflow.log_metric("perf/swag_total_seconds", swag_duration)
```

**Rationale**: MLflow is THE tracking store (CLAUDE.md). No parallel persistence.
Cloud Run MLflow server in europe-north1 is already configured.

### 3.2 Secondary: JSONL Artifact (per-job, machine-readable)

Extend the existing `timing/timing_report.jsonl` artifact (already generated by
`generate_timing_jsonl()` in `infrastructure_timing.py`):

```jsonl
{"phase":"setup","operation":"docker_pull","duration_seconds":480.2,"timestamp_utc":"2026-03-23T10:00:00Z"}
{"phase":"setup","operation":"dvc_pull","duration_seconds":120.5,"timestamp_utc":"2026-03-23T10:08:00Z"}
{"phase":"training","operation":"fold_0","duration_seconds":300.0,"epochs":2,"timestamp_utc":"2026-03-23T10:12:00Z"}
{"phase":"post_training","operation":"swag","duration_seconds":1500.0,"swag_epochs":2,"timestamp_utc":"2026-03-23T10:17:00Z"}
{"phase":"summary","operation":"job_total","duration_seconds":1800.0,"gpu_utilization":0.55}
```

### 3.3 Tertiary: DuckDB Analytics (cross-job analysis)

The existing `analytics.py` DuckDB engine queries MLflow runs. Add views:

```sql
-- Factorial performance overview
CREATE VIEW perf_factorial AS
SELECT
    t.model_family,
    t.loss_function,
    m."perf/setup_total_seconds" AS setup_s,
    m."perf/training_total_seconds" AS train_s,
    m."perf/swag_total_seconds" AS swag_s,
    m."perf/job_total_seconds" AS total_s,
    m."perf/gpu_utilization_fraction" AS gpu_util,
    m."cost/total_usd" AS cost_usd
FROM mlflow_runs r
JOIN mlflow_tags t ON r.run_id = t.run_id
JOIN mlflow_metrics m ON r.run_id = m.run_id
WHERE t.flow_name = 'train';
```

## 4. Optimization Opportunities

### 4.1 Docker Pull Optimization

**Current**: ~8-10 min to pull base image from GAR (same-region europe-north1).

| Optimization | Expected Speedup | Effort | Notes |
|-------------|-----------------|--------|-------|
| **Zstd compression** | 2-3x faster decompression | Low | `docker buildx build --output type=image,compression=zstd,compression-level=3` |
| **Layer caching on VM** | Skip pull entirely on retry | Low | SkyPilot spot recovery reuses VM disk if not preempted |
| **Image size reduction** | Proportional to size delta | Medium | Current base is ~8-12 GB. Audit unused packages. |
| **Pre-pull to GCE disk image** | Near-zero pull time | High | Custom GCE image with Docker image pre-cached |
| **Multi-arch slim image** | 30-50% size reduction | Medium | Strip debug symbols, use `--strip` in pip install |

**Recommended first step**: Zstd compression (low effort, high reward).

```bash
# Build with zstd (requires Docker Buildx)
docker buildx build \
  --output type=image,compression=zstd,compression-level=3 \
  -t europe-north1-docker.pkg.dev/minivess-mlops/minivess/base:latest \
  -f deployment/docker/Dockerfile.base .
```

**Measure**: Add timing to SkyPilot setup block:

```bash
# In train_factorial.yaml setup:
docker_pull_start=$(date +%s.%N)
# ... docker pull happens automatically via image_id
docker_pull_end=$(date +%s.%N)
```

Note: SkyPilot handles the Docker pull internally. The setup block timing captures
the overall setup duration. To isolate Docker pull time specifically, we need to
instrument the SkyPilot setup or parse SkyPilot logs post-hoc.

**Image size audit** (action item):
```bash
# Check current image size
docker images europe-north1-docker.pkg.dev/minivess-mlops/minivess/base:latest --format "{{.Size}}"
# Layer-by-layer analysis
docker history europe-north1-docker.pkg.dev/minivess-mlops/minivess/base:latest
# Dive tool for interactive analysis
dive europe-north1-docker.pkg.dev/minivess-mlops/minivess/base:latest
```

### 4.2 DVC Pull Optimization

**Current**: ~2-4 min to pull MiniVess data from GCS.

| Optimization | Expected Speedup | Effort | Notes |
|-------------|-----------------|--------|-------|
| **Partial pull** | 50% for debug runs | Low | `dvc pull data/raw/minivess -r gcs` already scoped |
| **GCS bucket same-region** | Already optimal | N/A | Both bucket and VMs in europe-north1 |
| **Pre-cached data disk** | Skip pull entirely | High | GCE persistent disk with data, attach to VM |
| **file_mounts with GCS** | Near-zero setup | Medium | SkyPilot `file_mounts` with `gs://` source |

**Recommended**: SkyPilot `file_mounts` for data (bypasses DVC entirely on cloud):

```yaml
# In train_factorial.yaml — replace DVC pull with direct GCS mount
file_mounts:
  /app/checkpoints:
    source: gs://minivess-mlops-checkpoints
    mode: MOUNT_CACHED
  /app/data/raw/minivess:
    source: gs://minivess-mlops-dvc-data/files/md5  # DVC cache structure
    mode: MOUNT  # Read-only, cached locally
```

**Caveat**: DVC stores files by content hash, not by filename. Direct GCS mount
would require either (a) restructuring the GCS bucket, or (b) a thin DVC-to-GCS
resolver. The DVC pull approach is simpler and already works. Optimization effort
should focus on SWAG first (10x larger time savings).

### 4.3 SWAG Overhead Reduction (HIGHEST PRIORITY)

**Current**: SWAG takes ~25 min for a 2-epoch debug run. Training takes ~5 min.
Overhead ratio: 5.0x.

**Why SWAG is 5x slower than training in the debug case**:

1. **Model rebuild**: `_run_swag_post_training()` rebuilds the model and DataLoader
   from scratch (lines 1121-1182 of train_flow.py). This includes re-loading splits,
   rebuilding transforms, and re-initializing MONAI CacheDataset.

2. **swa_epochs defaults to 10**: The `SWAGPluginConfig.swa_epochs` default is 10.
   For a 2-epoch debug run, SWAG runs 10 additional epochs (5x the training).
   For a 50-epoch production run, SWAG would add 10 epochs (0.2x overhead).

3. **No caching**: `cache_rate=0.0` in the SWAG DataLoader rebuild (line 1181),
   so every volume is loaded from disk on every epoch.

4. **BatchNorm update**: Full forward pass through training set for BN statistics.

**Optimization strategies (ordered by impact)**:

| Optimization | Expected Improvement | Effort |
|-------------|---------------------|--------|
| **Scale swa_epochs to training epochs** | 5x→2x for debug | Low |
| **Reuse DataLoader from training** | Save ~2-3 min rebuild | Medium |
| **Enable cache_rate for SWAG** | 30-50% per-epoch speedup | Low |
| **Skip BN update for debug** | Save 1-2 min | Low |
| **Parallel SWAG collection** | N/A (inherently sequential) | N/A |

**Recommended fix** (scale swa_epochs):

The SWAG swa_epochs should scale with training epochs. In
`run_factorial_post_training()` (line 539 of post_training_flow.py), the config
hardcodes `"swa_epochs": 2`. In the SkyPilot flow path (`_run_swag_post_training()`),
the default from `SWAGPluginConfig` is 10.

Proposal: SWAG epochs = `min(swa_epochs_config, max(2, max_epochs // 5))`.
- Debug (2 epochs): SWAG = 2 epochs (1.0x, not 5.0x)
- Production (50 epochs): SWAG = 10 epochs (0.2x)

```python
# In _run_swag_post_training() or SWAGPlugin.execute():
max_epochs = config.get("max_epochs", 50)
swa_epochs = swag_config.get("swa_epochs", 10)
effective_swa_epochs = min(swa_epochs, max(2, max_epochs // 5))
```

**Expected time savings**:
- Debug: SWAG drops from ~25 min to ~5 min (5x reduction)
- Per-job total: ~30 min → ~10 min (3x faster)
- Debug factorial 34 jobs: ~20 hrs → ~6-7 hrs (with RunPod controller)
  or ~4 hrs (with GCP controller, parallel submission)

### 4.4 Submission Latency Optimization

**Current**: ~36 min/job with RunPod controller (serial submission + cross-cloud API).

| Optimization | Expected Improvement | Effort |
|-------------|---------------------|--------|
| **GCP controller** | 36 min → ~5 min submission | Low |
| **Parallel job submission** | Linear speedup | Already implemented |
| **Batch job submission** | Reduce API overhead | Medium |

**GCP controller** is the single most impactful infrastructure change.
RunPod controller adds cross-cloud latency for every SkyPilot API call.
GCP controller + GCP VMs = same-cloud, same-region:

```bash
# One-time: migrate SkyPilot controller to GCP
sky check gcp
# Controller automatically provisions in the cheapest region
```

**Expected improvement**: Submission latency drops from ~36 min to ~5 min per job.
For 34 debug jobs: saves ~17 hrs of pure overhead.

### 4.5 Spot Provisioning Optimization

**Current**: ~10-15 min from STARTING to RUNNING.

This includes:
1. GCE spot VM allocation (~2-3 min)
2. Docker image pull (~8-10 min) ← already covered in 4.1
3. SkyPilot setup execution (~2-3 min)

The only controllable factors are Docker pull (4.1) and setup script efficiency.
Spot VM allocation is GCP-side and cannot be optimized.

These tests enforce performance budgets. They fail if measured or estimated
durations exceed budgets, catching regressions early.

```
tests/v2/unit/observability/test_perf_budgets.py
```

| Test Name | Budget | Rationale |
|-----------|--------|-----------|
| `test_swag_overhead_ratio_budget` | `swag_total / training_total <= 2.0` | SWAG should not exceed 2x training time |

| `test_gpu_utilization_minimum` | `gpu_active / job_total >= 0.50` | GPU should be active >50% of job time |
| `test_image_size_budget` | Base image < 15 GB compressed | Prevent image bloat |
| `test_amortization_budget` | `epochs_to_amortize <= 200` | Setup cost amortized within reason |

## 7. Production Factorial Time Estimate

### 7.1 Current Architecture (No Optimizations)

Production: 4 models x 4 losses x 2 aux_calib x 3 folds = 96 GPU jobs
Plus 6 zero-shot baselines = 102 total jobs.

Per-job estimate (50 epochs, DynUNet):
- Setup: 14 min
- Training: ~125 min (50 epochs x 2.5 min/epoch)
- SWAG: ~25 min (10 SWAG epochs x 2.5 min/epoch)
- Total: ~164 min (~2.75 hrs)

Total for 102 jobs (serial, RunPod controller):
- Job time: 102 x 164 min = 16,728 min = ~279 hrs
- Submission overhead: 102 x 36 min = 3,672 min = ~61 hrs
- **Total: ~340 hrs (~14 days)**

Cost at GCP L4 spot ($0.24/hr):
- GPU hours: 279 hrs
- **Cost: ~$67**

### 7.2 With Optimizations (Phase 1-4)

Per-job estimate after optimizations:
- Setup: 5 min (zstd docker, GCP controller)
- Training: ~125 min (unchanged)
- SWAG: ~25 min (10 epochs, already reasonable at 50 training epochs)
- Total: ~155 min (~2.58 hrs)

Total for 102 jobs (GCP controller, parallel submission):
- Job time: 102 x 155 min = 15,810 min = ~264 hrs
- Submission overhead: ~5 min (parallel, same-cloud)
- **Total: ~264 hrs (~11 days)** (but parallel execution available)

With SkyPilot parallel execution (e.g., 10 concurrent spot VMs):
- Effective wall time: 264 / 10 = ~26.4 hrs ≈ **~1.1 days**
- Cost: same ~$63 (parallel doesn't change GPU hours)

### 7.3 Debug Factorial After Optimizations

Debug: 32 GPU jobs + 2 zero-shot = 34 jobs.

Per-job (2 epochs, optimized SWAG):
- Setup: 5 min
- Training: 5 min
- SWAG: 5 min (2 SWAG epochs, not 10)
- Total: 15 min

Total for 34 jobs (GCP controller, parallel):
- Job time: 34 x 15 min = 510 min = 8.5 hrs
- With 10 concurrent VMs: 510 / 10 = 51 min ≈ **<1 hour**
- **Target of <4 hours easily achieved even with 3 concurrent VMs**

Cost at GCP L4 spot ($0.24/hr):
- GPU hours: 8.5 hrs
- **Cost: ~$2.04**

### 7.4 Model-Specific Variation

Different models have very different per-epoch times:

| Model | Est. Epoch Time | 50-Epoch Job | Notes |
|-------|----------------|-------------|-------|
| DynUNet | ~2.5 min | ~2.75 hrs | Fastest, most GPU-efficient |
| MambaVesselNet | ~3.5 min | ~3.6 hrs | CUDA kernel compilation overhead |
| SAM3 TopoLoRA | ~8 min | ~7.5 hrs | ViT-32L encoder, needs L4/A100 |
| SAM3 Hybrid | ~12 min | ~10.8 hrs | Largest model (6.65 GiB weights) |

SAM3 models dominate total cost. 32 SAM3 jobs x ~9 hrs avg = 288 GPU hrs = ~$69.
64 DynUNet+Mamba jobs x ~3 hrs avg = 192 GPU hrs = ~$46.
**Total production estimate: ~$115 across all 96 training jobs**.
