## The Rule

`import re` and regex patterns are **banned** for parsing structured data.

| Banned | Use Instead |
|--------|-------------|
| `re.match/search/compile` on Python source | `ast.parse()` + `ast.walk()` |
| `re.match/search` on YAML/JSON/TOML | `yaml.safe_load()`, `json.loads()`, `tomllib` |
| `re.match/search` on log lines | Emit as JSON (JSONL), parse with `json.loads()` |
| `re.match/search` on metric/param names | `str.split()`, `str.rsplit()`, `str.partition()` |
| `re.match/search` on file paths | `pathlib.Path` attributes (`.stem`, `.suffix`, `.parts`) |
| `re.match/search` on config keys | Proper dict access, `str.startswith()`, `str.split()` |

"Regex is sufficient" is a banned phrase. Claude does not get to self-assess.

## Why the Ban Exists (from foundation-PLR VIOLATION-001)

Regex fails on:
- Constants inside string literals (matches comments in docstrings)
- Multi-line assignments
- Type annotations
- Unicode characters in values
- Any format variation the author didn't anticipate

More importantly: **the ban enforces discipline**. When you reach for regex it means
you haven't defined a proper schema. If a string needs regex to parse, it should be
JSON/YAML instead.

## For GPU/Training Log Lines Specifically

The plan at `docs/planning/gpu-params-logging.md` originally proposed a regex for
parsing the GPU log line. This was removed. The correct approach:

- Human-readable line → printed to terminal (logger.info)
- Machine-parseable → emit a JSONL line to a structured log file, or store the
  `ResourceSnapshot` dataclass directly and serialize with `dataclasses.asdict()` +
  `json.dumps()`. Parse with `json.loads()`.

No regex needed if the data source is JSON.
